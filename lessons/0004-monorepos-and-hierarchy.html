01 // First Principles

How Agents Search the Directory Tree

Hello. When an AI agent performs tasks in your workspace, it needs to find context and rules. It does not just look in the workspace root folder. It searches your directory structure dynamically.

Let us explain this using first principles. An agent always works from a current directory. When it starts a task or runs a command:

  1. It checks the current working folder for an AGENTS.md file.
  2. If it does not find one, it moves up to the parent directory and checks again.
  3. It repeats this process, walking up the tree, until it reaches the root of your workspace.
workspace-root/ // Stop here (checks third)
├── AGENTS.md
└── apps/ // Checks second
└── web-app/ // Checks first (agent working folder)
├── AGENTS.md
└── src/
└── main.ts <-- Agent edits this file

By walking up the directory tree, the agent reads all instructions along the path. It combines them to build its final instruction set.

02 // Specificity and Overrides

Subdirectory Rules Override Root Rules

Sometimes, you might have conflicting instructions in different directories.

In these cases, the rule of specificity applies: a rule in a subdirectory AGENTS.md overrides a conflicting rule in a parent or root AGENTS.md.

This is extremely useful. It lets you write general rules for your whole workspace while keeping project-specific commands and rules separate.

The Override Process

Imagine your root AGENTS.md contains:

./AGENTS.md (Root) 
## Build & Test Commands
`npm run build`

But you have a Python service in ./services/backend/ with its own AGENTS.md:

./services/backend/AGENTS.md 
## Build & Test Commands
`poetry run pytest`

When the agent works inside ./services/backend/, it uses `poetry run pytest` to verify its changes instead of the root build command. The local file overrides the root command completely.

03 // Monorepos

Best Practices for Monorepo Hierarchy

In a monorepo setup, you have multiple sub-projects inside one repository. If you put all rules for every service in a single root file, the file will become massive and confusing.

Use these rules to design a clean hierarchy:

✅ Clean Hierarchy

How to structure rules

  • Put global rules in the root file (such as your username, styling principles, or off-limit paths).
  • Put service-specific build and test commands in each service subdirectory.
  • Define framework-specific style guides (like React vs NestJS) in the subdirectory where they are used.
❌ Bad Practices

What to avoid

  • Do not copy and paste the same full file into every folder. That makes updates painful.
  • Do not write frontend rules in the backend directory. Keep them separated.
  • Do not force one global test command if services use different build tools.

Question 1 of 3

How do AI agents search for AGENTS.md instruction files in a workspace?

🎉 Correct! The agent searches from its current location and walks up parent folders. This lets it read both local and global rules.
❌ Incorrect. The agent does not scan the whole hard drive or look only at the OS root. It searches up from its current working directory to the workspace root.
Question 2 of 3

If a root AGENTS.md and a subdirectory AGENTS.md have conflicting build instructions, which one will the agent use in that subdirectory?

🎉 Correct! In hierarchical file setups, specific instructions in nested folders override conflicting rules from parent directories.
❌ Incorrect. The agent will not crash or choose randomly. The rule of specificity means subdirectory instructions override parent ones.
Question 3 of 3

What is the best way to manage AGENTS.md files in a monorepo?

🎉 Correct! Storing global rules at the root and specific instructions in subdirectory files keeps the workspace clean and keeps files small.
❌ Incorrect. Putting all instructions in the root makes it messy, and copying the exact same file is hard to maintain. Storing global rules at the root and app-specific rules in subdirectories is the best approach.
