What is AGENTS.md, and Why Does It Exist?

The foundation. Understand the problem it solves before you write a single line.

~10 min read 3 quiz questions Beginner

The Problem

Every time you start a new session, the agent forgets everything

AI coding agents like Antigravity, Cursor, Copilot, and Claude Code are powerful - but by default they know nothing about your project. Every new session is a blank slate.

This means you end up repeating yourself. You type things like:

  • ✗ "By the way, we use TypeScript strict mode..."
  • ✗ "Don't touch the /vendor folder..."
  • ✗ "We always run npm test before committing..."
  • ✗ "Our naming convention is camelCase for functions, PascalCase for classes..."

This is friction. It wastes your time and the agent gets it wrong because you forgot to mention it.

💡
The core idea

AGENTS.md is a file you put in your project that answers all these "by the way" questions upfront - permanently. The agent reads it at the start of every session.

What It Is

AGENTS.md is a README for machines

You already know what a README.md is - it is a file that explains your project to humans. It covers what the project does, how to install it, and how to contribute.

AGENTS.md does the same thing, but for AI agents. It is written in plain Markdown and lives in the root of your repository. When an agent starts working on your project, it reads this file first.

👤 README.md

For humans

  • What this project does
  • Installation steps
  • Contribution guidelines
  • License information
  • Screenshots, demos
🤖 AGENTS.md

For AI agents

  • Exact build & test commands
  • Code style rules to follow
  • Files to never touch
  • How to run the test suite
  • Project-specific conventions

They are kept separate by design. You do not want to bloat your README with machine-specific instructions that confuse human contributors.

✅
Used by 60,000+ open-source projects

AGENTS.md is an open standard - not owned by any company. It works with Antigravity, Cursor, Copilot, Aider, Windsurf, Zed, Devin, and many more. (Note: Claude Code uses a similar file named CLAUDE.md). Source: agents.md

Compatible Tools

One file, works across all major agents

Write it once. Every agent that supports AGENTS.md will read it automatically:

🤖 Antigravity ▶ Cursor 🥇 GitHub Copilot 🌐 Claude Code (uses CLAUDE.md) 🔭 OpenAI Codex 🌍 Google Jules 💻 VS Code (Copilot Agent) 🚀 Windsurf 🆕 Aider 🦵 Devin ⚡ Zed 🐧 Goose
How It Works

The agent reads it before it does anything

Here is the order of events when you start a session with an AI coding agent that supports AGENTS.md:

1. You open your project and ask the agent something

"Add a dark mode toggle to the settings page"

2. The agent looks for AGENTS.md in the project root

If it finds one, it reads every instruction in it - before doing anything else

3. The agent now knows your rules

It knows your code style, your test commands, what files to avoid, and your naming conventions

4. The agent gets to work - correctly, first time

No "by the way" needed. It follows your project rules automatically

⚠️
Important: Monorepos have a hierarchy

In a monorepo (one big repo with multiple sub-projects), you can have an AGENTS.md in the root AND in subdirectories. The agent uses the file closest to where it is working. We will cover this in Lesson 4.

First Look

What does one actually look like?

Below is the simplest possible AGENTS.md - a minimal but real example:

AGENTS.md MARKDOWN 
# AGENTS.md

## Setup commands
- Install deps: `pnpm install`
- Start dev server: `pnpm dev`
- Run tests: `pnpm test`

## Code style
- TypeScript strict mode
- Single quotes, no semicolons
- Functional patterns where possible

## Never touch these
- /vendor/ and /generated/ directories
- Any file ending in *.generated.ts

That is it. Plain Markdown. No special syntax. No YAML frontmatter. Just clear instructions organised into sections.

Anatomy of the example above
# AGENTS.md
The file title - tells the agent (and humans) what this file is for
## Setup commands
A section heading - agents read sections like chapters, so label them clearly
- Run tests: `pnpm test`
Exact command in backticks - always use the exact runnable command, not a description of it
- TypeScript strict mode
A code rule - only rules that deviate from the default or can't be inferred from the code
/vendor/ and /generated/
Protected paths - the most important constraint to define clearly - what to never modify

Quick Check

Question 1 of 3

What is the main problem that AGENTS.md is designed to solve?

🎉 Exactly right. AGENTS.md gives the agent information it can't infer from the code - things you would otherwise repeat at the start of every session.
❌ Not quite. AGENTS.md complements README.md - it does not replace it. And while it can reduce the need for tool-specific files, the core purpose is giving agents persistent project context.
Question 2 of 3

You are starting a new project. Where should AGENTS.md be placed?

🎉 Correct. The root of the repo is the standard, predictable place. Agents know to look there first. In monorepos you can also add them to subdirectories for scoped rules.
❌ Not quite. AGENTS.md belongs in the repository root - the same level as README.md. That is the standard place agents are built to look for it.
Question 3 of 3

Which of these belongs in AGENTS.md?

🎉 Perfect. Exact runnable commands are the most valuable thing to put in AGENTS.md. Option A belongs in README, Option B is already enforced by tooling so it's redundant, Option D is redefining the agent's core persona - which AGENTS.md is not for.
❌ Not quite. The key principle is: only include things the agent cannot infer from the codebase or that aren't already enforced by your tooling. Exact commands are perfect examples. Project descriptions belong in README. Persona redefinition is off-scope.
