Lesson 3: Writing Your First AGENTS.md File

A step-by-step hands-on guide for you to build an AGENTS.md file from scratch in any codebase.

~15 min read 3 quiz questions Hands-on
System Status Lesson 3 / 7
01 // First Principles

How an Agent Sees Your Code

Hello, User. To build a great AGENTS.md file, you must first understand how an AI agent operates. An AI agent is a program that reads files, decides on actions, and runs commands. It does not have human memory or project context when it starts a new session. It only knows what is in the codebase.

If you do not write down your instructions, the agent must guess. It might use the wrong tool, change the wrong files, or write code in a style you do not want. We write the AGENTS.md file to stop this guessing.

Let us build your first AGENTS.md file from scratch. Follow these four steps.

02 // Step One

Step 1: Find and Test the Build Command

An AI agent must be able to check if its code edits work. To do this, it needs to run your project build and test commands. We must find the exact command and write it down.

Look at your project root files.

  • In a JavaScript project, open `package.json` and look at the `scripts` section.
  • In a Python project, check `pyproject.toml`, `Makefile`, or `requirements.txt`.
  • In a Rust project, you will likely use Cargo commands.
⚡
Action for you:

Run the test or build command in your terminal before you write it down. Make sure it runs to completion with no errors. If the command fails for you, it will fail for the agent too.

Once you find the correct command, write it under a header named `## Build & Test Commands`. You must wrap the command in backticks. Backticks tell the agent that the text is a runnable command, not just general description.

Drafting AGENTS.md 
## Build & Test Commands

Run the build to verify TypeScript compiles:
`npm run build`

Run the unit tests:
`npm test`
03 // Step Two

Step 2: Mark Protected Paths as Off-Limits

AI agents work very fast. They can search the workspace and edit multiple files in seconds. Sometimes, they might try to change files that they should not touch.

Identify the folders in your workspace that are:

  • Auto-generated: Folders like `dist/`, `build/`, `out/`, or `public/bundle/`. If the agent edits these, the compiler will just overwrite the changes.
  • Third-party dependencies: Folders like `node_modules/`, `vendor/`, or `.venv/`. Changing these directly will break package updates.
  • Sensitive configurations: Special deployment scripts or environment files.
⚠️
Protecting your files

Write down these off-limit paths clearly. Tell the agent that it must never read, write, or change files in these paths. This keeps your build safe and clean.

Create a section in your AGENTS.md named `## Constraints` or `## Protected Paths`. List these paths clearly.

Drafting AGENTS.md 
## Protected Paths

Do NOT read or modify any files in the following folders:
- `dist/` - this contains compiled production code
- `vendor/` - this contains third-party dependencies
04 // Step Three

Step 3: Add Design Rules That Linters Cannot Catch

Remember the "Focus on the Unknown" principle from Lesson 2. Your agent can already read your config files, like `.eslintrc` or `tsconfig.json`. You do not need to write rules that are already enforced by your linter.

Instead, write style rules about your custom design choices. Ask yourself: "What would I tell a new developer on their first day?"

✅ Write Custom Patterns

Design Guidelines

  • "Use the Fetch API instead of Axios."
  • "Always write tests in the same folder as the component."
  • "Use functional components for all new UI."
❌ Avoid Linter Duplicates

Redundant Rules

  • "Use 2 spaces for indentation." (Linter does this)
  • "Do not leave unused variables." (Linter does this)
  • "This is a React app." (Agent finds this in configs)

Write these guidelines under a header named `## Code Style Guidelines`. Use simple bullet points. Keep each point short and easy to understand.

Drafting AGENTS.md 
## Code Style Guidelines

- Prefer native browser APIs over external utility packages.
- Group business logic inside hook files rather than UI files.
05 // Step Four

Step 4: Create and Test Your AGENTS.md File

Now, you are ready to create the file.

  1. Create a new file called `AGENTS.md` at the root of your project workspace.
  2. Copy your sections (Build & Test, Protected Paths, and Code Style) into the file.
  3. Save the file.

After creating the file, you should test how the agent reacts. Ask the agent to build the project. It should read your `AGENTS.md` file, locate the test command in backticks, and run it. If it does not, review your file formatting. Make sure the headers and backticks are clean.

Question 1 of 3

Hello. What is the most basic truth about how an AI agent gets its instructions?

🎉 Correct! An AI agent uses files in the workspace as its source of truth. Setting up AGENTS.md stops the agent from guessing.
❌ Incorrect. The agent does not have intuition or access to your thoughts. It reads configuration files and workspace instructions to understand rules.
Question 2 of 3

Why should you write executable build and test commands inside backticks in your AGENTS.md file?

🎉 Correct! Putting commands in backticks makes it easy for the agent to find and run them directly in the terminal.
❌ Incorrect. Backticks do not hide commands or change terminal behavior. They format the text as code, which helps the agent identify executable actions.
Question 3 of 3

Which of these is a good rule to put in the Code Style section of your AGENTS.md file?

🎉 Correct! This is a custom architectural rule that linters cannot check automatically. It is a perfect fit for AGENTS.md.
❌ Incorrect. Indentation is checked by linters. "Write comments" and "Do not write bugs" are too generic to help an agent write correct code.
