01 // First Principles

Why Advanced Rules Matter

Hello, 54b4rn4th. To master agent coordination, we must look at how an agent reads instructions. An agent is a tool that writes code based on context. It knows basic programming syntax, but it does not know your personal goals or design decisions.

Many developers think that a code linter is enough to keep a codebase clean. This is false. A linter checks basic syntax formatting, like missing semicolons or spaces. A linter cannot check if a test matches your mock files. It cannot check if the agent selected a bad library. It cannot check if a git commit message is clear.

We write advanced rules in our instructions to solve this problem. These rules direct the agent to make design and testing choices that match your expectations.

02 // Custom Style Guidelines

Rules Linters Cannot Catch

An agent will write code that runs, but the code may not fit your architecture. You must write rules for architectural patterns and library selections.

For example, if you do not want the agent to use external HTTP clients like Axios, you must write a rule. If you want the agent to write components using a specific file structure, you must explain that structure.

✅ Write Architecture Rules

Design Guidelines

  • "Use native browser fetch instead of external packages."
  • "Put component logic in hooks files to keep UI files simple."
  • "Use CSS variables instead of inline styles."
❌ Avoid Formatting Duplicates

Redundant Rules

  • "Use tabs for formatting." (Linters do this)
  • "Add a semicolon at the end of each statement." (Linters do this)
  • "Export every function." (Syntax checks do this)

Write these design rules under a header in your instructions file. Focus on code structure and library limits.

Drafting AGENTS.md 
## Architecture & Library Rules

- Use native Web APIs whenever possible.
- Avoid installing new dependencies without asking first.
- Split files if they grow larger than 200 lines of code.
03 // Testing Strategies

Testing Rules and Strategies

When an agent edits code, it must verify its changes. If you do not specify how to write and run tests, the agent will write tests that do not match your testing pattern.

You must instruct the agent on where test files live, how to name them, and what tool to use to run them.

🧪
Action for 54b4rn4th:

Tell the agent exactly what to test. For example, state that every new utility function must have a unit test file in the same folder as the utility file.

Create a testing rule block that contains the test runner command and file naming rules:

Drafting AGENTS.md 
## Testing Rules

- Put test files in the same folder as the source file.
- Name test files `[filename].test.ts`.
- Run tests using `npm run test` after making any edits.
04 // Mock Guidelines

Mocking Guidelines

Mocks are fake objects that simulate real services. Mocks help run tests quickly without calling live external databases or network links.

If an agent writes custom mocks for every test, it will create fragile tests. Fragile tests break when you make simple changes to the project.

To prevent fragile tests, write clear rules about mock styles:

  • State which libraries the agent must use for mocking (for example, Mock Service Worker for API calls).
  • Tell the agent to clean up mock states after each test run.
  • Prevent the agent from mocking internal helper functions. Mocks should only be used for external boundaries.
Drafting AGENTS.md 
## Mocking Guidelines

- Use Mock Service Worker (MSW) for API mock tests. Do not mock fetch manually.
- Reset mock handlers after each test using `afterEach(() => server.resetHandlers())`.
- Do not mock internal business logic or helper utilities.
05 // PR and Commit Templates

Automating Commits and Pull Requests

An agent works fast and makes many changes. If the agent makes commits with unclear messages, human developers cannot review the project history.

To fix this, write rules that force the agent to use a template for git commits and Pull Requests (PRs). Use the Conventional Commits format to make changes easy to read.

📝
Structuring Commits:

A template tells the agent to list the issue title, the files modified, and the test results. This makes it easy for you to review and merge changes.

Drafting AGENTS.md 
## Commit & PR Guidelines

Use this commit format:
`[type]: [short summary]`

Types:
- `feat` (new features)
- `fix` (bug fixes)
- `test` (adding tests)

PR descriptions must include:
1. Problem description
2. Solution description
3. Test commands executed
