First Principles

Why do we structure AGENTS.md?

An AI agent needs structured instructions to perform tasks. Without clear categories, the agent can get confused or miss critical rules. By structuring your AGENTS.md into specific pillars, you give the agent a clear way to understand your codebase rules.

This structure is built from the ground up to address how agents search, compile, and edit. Let us break down the structure into its core parts.

The Three Pillars

The 3 Pillars of AGENTS.md

Every great AGENTS.md file is built on three core pillars. Each pillar addresses a specific risk or requirement for the agent.

Pillar A

Build & Test Commands

What it is: The exact, runnable commands to build, check, and test the project.

Why it is needed: Every project has a unique setup (npm, pnpm, yarn, cargo, poetry). If you do not give the exact command, the agent might guess. This can lead to package issues, broken environments, or wasted runs.

Example: Use backticks to specify commands exactly: `pnpm test` or `npm run build`.

Pillar B

Code Style Guidelines

What it is: Custom rules and architecture choices that linting tools cannot catch.

Why it is needed: Automated tools like ESLint or Prettier catch basic syntax. But they cannot catch design patterns (like "prefer functional components") or custom helpers. Focus only on rules that your tooling does not enforce.

Example: "Use snake_case for database fields, but camelCase in TypeScript."

Pillar C

Constraints / Off-limits

What it is: Protected directories, files, or paths that the agent must never edit.

Why it is needed: AI agents work very fast and can change many files. Some folders are automatically generated (like /dist or /generated) or contain third-party packages (like /vendor). If an agent edits these files, it can break your build or override machine-generated code.

Example: "Never touch files in /vendor or /generated."

Design Principle

Focus on the Unknown

The most common mistake when writing an AGENTS.md file is listing everything about the project. This is redundant.

Coding agents can read. When an agent enters your project, it reads config files like package.json, tsconfig.json, and .eslintrc. It can look at your files and see that you are using React or TypeScript.

💡
The "Focus on the Unknown" Rule

Only write rules that the agent cannot find by reading the codebase. Do not write duplicate info that is already defined in config files.

✅ What to write

Unseen Rules

  • Build commands: `pnpm test`
  • Protected folders: /vendor
  • Architectural choices
  • Custom patterns
❌ What to leave out

Redundant Info

  • "This is a React app" (Found in configs)
  • "Use semicolons" (Enforced by linter)
  • List of standard packages used
  • General Git tutorials

Question 1 of 3

Which of these should go in the 'Build & Test' section of your AGENTS.md file?

🎉 Correct. Providing exact, runnable commands in backticks stops the agent from guessing and avoids environment issues.
❌ Incorrect. The agent needs exact commands that it can run directly. Explanations, historical lists, or links do not give the agent the runnable tools it needs.
Question 2 of 3

What is the 'Focus on the Unknown' principle?

🎉 Correct. The agent can read existing configuration files. You should only list guidelines and rules that are not already present in the codebase.
❌ Incorrect. The principle states that you should not repeat rules that are already defined in code, configs, or linters. You should not hide files or leave important commands empty.
Question 3 of 3

What is the most critical constraint to define in an AGENTS.md?

🎉 Correct. Defining protected paths and files prevents the agent from overwriting generated code or third-party files.
❌ Incorrect. Defining off-limit folders (like vendor or generated files) is the most critical constraint. This protects your workspace from broken builds caused by agents overwriting dependencies.
