01 // First Principles

Why One Agent is Not Enough

Hello, 54b4rn4th. To understand multi-agent systems, we must look at the physical limits of how AI models process information. We can build our understanding from three basic facts.

Fact 1: The Context Window is Limited

An AI agent has a set limit on how much text it can read and hold in memory at one time. This includes the rules of the project, all files it has read, the chat history, and the outputs of commands. In a large project, these files and logs quickly fill up the memory. When this happens, the agent starts to forget earlier instructions or slows down.

Fact 2: Generality Breeds Errors

If a single agent has to run tests, write database queries, design CSS stylesheets, and write project documentation all at once, its focus is split. It has to keep hundreds of rules active in its memory. This split focus leads to minor mistakes, skipped checks, and rules being ignored.

Fact 3: Smaller Tasks Need Less Context

If we extract a single task - like finding a bug in a database query - we do not need to know about the frontend styles or the build scripts. We only need the database files. By giving this small task to a clean, separate agent session with only the database files, the agent works faster, costs less, and makes fewer mistakes.

💡
First Principles Definition

Multi-agent orchestration is the act of splitting one complex project goal into small, isolated tasks. A lead agent (orchestrator) directs the work, while specialized worker agents (subagents) carry out the single tasks.

02 // Local Boundaries

Scoped AGENTS.md Files

When the lead agent creates a worker agent to solve a task, how does the worker know the rules for that specific part of the code? This is where scoped AGENTS.md files come in.

Instead of having a single, massive AGENTS.md file at the root of your project that contains every rule for every folder, you can place smaller, scoped AGENTS.md files inside subdirectories.

Project Directory Structure
my-project/
├── AGENTS.md                 # Root file - global build rules and general info
├── frontend/
│   ├── AGENTS.md             # Scoped file - rules for CSS, React, and layout styling
│   └── main.jsx
└── backend/
    ├── AGENTS.md             # Scoped file - rules for SQL, database keys, and API safety
    └── server.js

How Worker Agents Use Scoped Files

When an agent starts a task in a specific directory (like the backend/ folder), it reads the AGENTS.md file inside that folder. This local file acts as the source of truth for that worker.

This approach has three major benefits:

  • No Distractions: The backend worker agent does not read frontend CSS rules. This saves context window space.
  • Custom Build Commands: The backend worker can have a specific run command (like npm run test:backend) that is different from the global build command.
  • Cleaner Code: You do not have to update a single huge file every time you change a local rule.
03 // Instructions

Designing Role-Based Instructions

To make worker agents work well, you must define clear roles. If you tell a worker agent "fix the bugs in the backend code", it is too vague. The agent will guess what to do.

Instead, design role-based instructions using four key parts:

  1. Role Name: Give the agent a job title (like Database Optimizer or API Tester). This sets the tone for how it acts.
  2. Clear Goal: State the exact problem the worker must solve.
  3. Tools and Scope: Explicitly state which folders the worker can read and write to, and what commands it can run.
  4. Boundaries: Tell the agent what it is NOT allowed to do (like modifying database schemas or changing main packages).
Poor Instruction

Generic & Unbounded

  • Role: Helper Agent
  • Task: "Fix the slow database code."
  • No list of permitted tools or commands.
  • No limits on which files can be edited.
  • Result: The agent might refactor the entire database schema, breaking other backend features.
Good Instruction

Role-Based & Scoped

  • Role: Database Optimizer
  • Task: "Find slow queries in backend/db/ and add indexes to speed them up."
  • Scope: "Only edit backend/db/queries.js. Do not edit database tables or schemas."
  • Command: Use npm run db:benchmark to test query speed.
  • Result: The agent solves the speed problem safely without touching other parts of the project.

Question 1 of 3

Why should you divide a large programming task among multiple specialized agents instead of using a single agent?

🎉 Correct! Breaking down a task into specialized steps stops the agent from getting distracted and helps it fit all key information inside its context window.
❌ Incorrect. Parallel execution does not speed up the local compiler. The primary reason is to manage the context window and maintain focus on a single task.
Question 2 of 3

How does a scoped AGENTS.md file inside a subdirectory help a worker agent?

🎉 Correct! Scoped AGENTS.md files contain local instructions that only apply to the worker's subdirectory, preventing context waste.
❌ Incorrect. Scoped files do not delete files or run tests on remote servers. They provide local rules and commands that keep the worker focused.
Question 3 of 3

Which of the following is a vital element when designing role-based instructions for a worker agent?

🎉 Correct! Providing a clear role name, a specific task, and explicit file/tool boundaries keeps the agent safe and focused on its target goal.
❌ Incorrect. Generic instructions lead to agent distraction. Safe and successful tasks require clear role names, specific tasks, and clear file and tool boundaries.
